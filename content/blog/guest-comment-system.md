博客上线一段时间了，后台数据挺有意思：日均 UV 在涨，但评论区始终冷冷清清。一开始觉得是内容不够好，后来自己用 visitor 身份走了一遍流程——想评论？先注册。注册？填邮箱、设密码、收验证邮件……算了，关掉。

登录门槛确实把大部分表达欲挡在了门外。这很矛盾：我想看到更多反馈，但又不希望为了几条评论让访客走完一整套注册流程。于是决定加个功能：**游客评论**，不登录也能说话。

## 改了什么

说起来不复杂，但实际牵了 8 个文件。核心就两件事：数据库允许匿名写入，前端加个昵称输入框。

### 数据库

`post_comments` 表原来有个 `NOT NULL` 的 `author_id` 外键，指向 `auth.users`——这意味着每条评论必须绑定一个注册用户。改掉它：

```sql
alter table post_comments
  alter column author_id drop not null,
  add column guest_name varchar(100),
  add column ip varchar(45);
```

新增的 `guest_name` 存访客自己填的昵称，`ip` 用来做频率限制。同样的改动也用在留言板（`guestbook_messages`）上。

RLS 策略也要加一条：

```sql
create policy "guest_comment" on post_comments for insert with check (
  auth.role() = 'anon' and author_id is null and guest_name is not null
);
```

意思很简单：匿名用户可以插入，但必须提供昵称，`author_id` 必须为空（防止伪造 user ID）。

### 频率限制

游客没有账号封禁的概念，所以必须有替代的防滥用手段。我做了个最简单的 IP 频率限制：同一个 IP 一小时内最多发 3 条。实现就是一个工具函数：

```ts
// checkIpRateLimit(ip, 'post_comments', 3, 60)
// → { allowed: boolean, remaining: number }
```

查询 `post_comments` 表，按 `ip` 和 `created_at` 计数。超过就拒绝，返回剩余时间提示。

3 条一小时够不够？对正常访客来说绰绰有余，对刷子来说成本够高了。先这样，如果后面真有被刷的问题再收紧。

### 前端

登录用户看到的评论区没有变化。未登录用户会在评论框上方多一行昵称输入，默认值为"匿名游客"，不填也能发——但不能为空字符串。提交时把昵称和内容一起传给 Server Action，Server Action 检测到没有 user 就走游客分支：取 IP、查频率、写入 `guest_name` 和 `ip`。

游客评论在列表里没有头像链接（因为没有用户主页），显示昵称和一个灰底的「游客」标签。

### 没改的

- **评论点赞**：还是要求登录。游客能评论已经是开放了，点赞再加匿名的话，数据上很难区分真实互动和机器行为。
- **删除权限**：文章作者可以删任何评论，包括游客的——这条逻辑本来就支持。
- **500 字限制、嵌套回复、加载更多**：全部不变，因为改动都在数据来源层，组件逻辑不需要动。

## 一些思考

游客评论最怕的是什么？广告、色情、政治敏感。我的应对策略其实挺基础的：IP 频率限制 + RLS 兜底 + 文章作者人工删评。没有上验证码，没有关键词过滤——小站没必要，流量还没大到值得针对的程度。

另一个取舍是**登录用户的体验不能降级**。登录用户发表评论不需要填昵称、不经过频率限制、插入的数据和之前完全一样。游客支持是叠加在现有逻辑上的，不是替代。改动前后登录用户的评论流程零变化。

对了，评论区之前有个小问题——未登录时点「评论」按钮，会弹一个"请先登录后再评论"的提示。现在游客可以直接发了，这个提示自然也就改成了显示后端返回的真实错误信息。

这个功能上线后评论区会不会变热闹，说实话我也不知道。但从数据上看，注册转化率确实很低——100 个访客里可能不到 1 个会注册。哪怕游客评论转化率只有 5%，那也是 5 倍的互动量。
